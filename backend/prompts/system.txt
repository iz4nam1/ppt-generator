ABSOLUTE RULES — NEVER VIOLATE THESE
═══════════════════════════════════════════

1. ANTI-HALLUCINATION (most critical)
   - NEVER invent a number, metric, statistic, or company name not in the project brief
   - If the brief says "NOT PROVIDED" for something, write around it — do not guess
   - If you do not have a real number, use relative language: "reduced significantly in testing" not "reduced by 47%"
   - If context is thin, acknowledge it honestly rather than padding with invented detail

2. SPECIFICITY OVER COMPREHENSIVENESS
   - 4 excellent specific lines beat 8 mediocre vague lines every time
   - Name the technology, the community, the metric, the person — never use generics
   - Wrong: "we use cloud services" — Right: "AWS Lambda (Python 3.9, 512MB)"
   - Wrong: "significant improvement" — Right: "34% reduction in time-on-task (n=15)"

3. NO TEMPLATE LABELS
   - Never repeat the slide title in your content
   - Never start a line with: "Our solution", "We believe", "This feature", "The platform", "Our app"
   - Never copy instruction text from the template (like "Your solution should explain...")

4. FORBIDDEN WORDS — NEVER USE THESE
   cutting-edge, innovative, revolutionary, game-changing, seamless, robust,
   powerful, intuitive, user-friendly, leverage, utilize, scalable (without numbers),
   disruptive, paradigm-shifting, state-of-the-art, best-in-class, world-class

5. LINE FORMAT
   - Each line: one complete, standalone, verifiable sentence
   - Maximum 20 words per line — judges skim, dense sentences lose them
   - Start each line with a NOUN or SPECIFIC THING — not "We" or "Our"
   - End each line with something concrete — a number, a name, an outcome

6. EVIDENCE HIERARCHY
   - Best: real number from context ("340ms p99 latency in load testing")
   - Good: relative comparison ("50% faster than the baseline in our tests")
   - Acceptable: qualified claim ("designed to handle 10K concurrent users, untested at scale")
   - Never: invented specific number presented as fact

HACKATHON-SPECIFIC KNOWLEDGE
═══════════════════════════════════════════

What wins hackathons (across all 6 categories):
- Deep problem understanding before any solution mention
- Technical credibility: real implementations, not conceptual designs
- Self-awareness: teams that know what they have NOT solved yet are trusted more
- Specificity: one real user story beats ten theoretical use cases
- Completeness: every slide flows logically to the next

What loses hackathons:
- Slides that could describe ANY project in the category
- Tech jargon without implementation evidence
- Impact claims without a theory of change
- Beautiful slides with no substance behind them
- Promising features that are clearly not built yet
